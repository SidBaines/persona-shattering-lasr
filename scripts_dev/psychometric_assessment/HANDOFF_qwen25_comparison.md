# Handoff — cross-model psychometric comparison (Qwen2.5-7B on B rollouts)

This document is a self-contained brief for an agent picking up the
"administer psychometric questionnaires on new models and compare" work on
a fresh machine. Everything needed lives on GitHub (branch
`sid/psychometric-refactor`) and HuggingFace (`persona-shattering-lasr/psychometric-fa-runs`).

Last pushed commit: `1bc2d23 sid/psychometric-refactor add cross-model response + Tucker congruence analysis`.

## Research goal

The parent project studies how personality traits embed in LLMs. One
sub-project administers psychometric questionnaires (v5 Likert, trait_ocean_v1
ABCD) to the *same* set of diverse personas generated from Llama-3.1-8B
rollouts, but under *different* response-model backbones. We want to know:

1. How similar are marginal response distributions across models?
2. For the same persona (same rollout prefix), do different models answer similarly?
3. Do the latent factor structures (FA + Tucker's φ) agree across models?

"Personas" here = single-persona rollouts from the `B` preset: 2 500 prompts ×
1 rollout × 15 assistant turns, generated by Llama-3.1-8B-Instruct using
scenarios v2 and user-sim prompt v6. The `B` rollout cache lives on HF at
`rollouts-llama318binstruct-t1.0-15t-2500p-seed436-scenarios_v2-uprompt_v6`.

HF run-id suffixes:
- Llama (B, default): no cross-model tag (`…direct` for v5, `…direct-lp20` for trait_ocean)
- Qwen3-8B: `-resp_qwen38b` (naming done by the earlier agent)
- Qwen2.5-7B: `-qm_qwen257binstruct` (our naming)

Combined FA of `(v5, trait_ocean_v1)` for Qwen2.5 is uploaded to HF at
`combined/combined-R[B]-Q[v5+trait_ocean_v1]-qm_qwen257binstruct/`. Includes
oblimin + varimax PAF loadings (166 surviving items × 7 factors,
`fa_7_principal_{rotation}.npz`), parallel analysis, full Stage 5
validation artifacts (shuffle, item_holdout, stability_sweep, bootstrap,
split-half congruence, CV k-curve, k-sensitivity, persona-item CV, etc.).

**Headline findings so far:**

- v5 direct-gen (1–5 Likert): Qwen2.5 collapses toward "3" on ambiguous items.
  Per-item response std ~0.47 vs 0.74 Qwen3 vs 0.83 Llama. Persona-mean
  cross-model Spearman: Llama↔Qwen3 = 0.74, Llama↔Qwen2.5 = 0.27.
- trait_ocean_v1 logprob (ABCD, top_logprobs=20): **Qwen2.5 is bimodal**, not
  midpoint — 75% of prob mass lies at ≤0.1 or ≥0.9. Persona-mean
  Llama↔Qwen2.5 ρ = 0.39 (recovered from 0.27 under v5).
- So Qwen2.5's "personality expressiveness" is modality-dependent: hedge under
  Likert direct-gen, commit under single-token logprob. Llama is graded in
  both. This is a measurement-validity warning for the research.

## Quick-start on a new machine

```bash
git clone git@github.com:SidBaines/persona-shattering-lasr.git
cd persona-shattering-lasr
git checkout sid/psychometric-refactor
uv sync   # or: pip install -r requirements.txt

# Populate .env with at least: HF_TOKEN, OPENROUTER_API_KEY (for user-sim
# traffic if you re-generate rollouts), optionally ANTHROPIC_API_KEY.
cp .env.example .env   # if .example exists; otherwise create by hand
```

### 1. Hydrate existing artifacts from HF

Stage 2 (questionnaire) artifacts auto-hydrate when the main script runs —
no manual step needed. The combined Qwen2.5 FA is *not* auto-hydrated;
download it explicitly if you want to skip re-running FA:

```python
# One-off hydration for the Qwen2.5 combined FA dir.
from src_dev.unsupervised_runs.io import hydrate_dataset_subtree
from pathlib import Path
hydrate_dataset_subtree(
    repo_id="persona-shattering-lasr/psychometric-fa-runs",
    path_in_repo="combined/combined-R[B]-Q[v5+trait_ocean_v1]-qm_qwen257binstruct",
    local_dir=Path("scratch/psychometric_fa/combined-R[B]-Q[v5+trait_ocean_v1]-qm_qwen257binstruct"),
)
```

### 2. Produce Llama-on-B combined FA

The main pipeline driver is
`scripts_dev/unsupervised_embeddings/psychometric_rollout_fa.py`. To run
Llama's FA:

```python
# In the script, around the "Cross-model questionnaire config" block:
CROSS_MODEL_QUESTIONNAIRE = False   # ← flip to False for the Llama pass
```

Also confirm:

```python
PAIRS = [("B", "v5"), ("B", "trait_ocean_v1")]
STAGES_TO_RUN = ["rollouts", "questionnaire", "factor_analysis", "validation"]
```

Launch (tmux + unbuffered logging):

```bash
mkdir -p scratch/logs
LOG=scratch/logs/psychometric_llama_on_B_fa_$(date +%Y%m%d_%H%M%S).log
tmux new-session -d -s psych_llama \
  "cd $(pwd) && PYTHONUNBUFFERED=1 uv run python -u -m scripts_dev.unsupervised_embeddings.psychometric_rollout_fa 2>&1 | tee $LOG"
```

- Stage 1 and Stage 2 will hydrate from HF (no inference — fast).
- Stage 3 and 5 will run on CPU; ~15–25 min.
- Output lands at `scratch/psychometric_fa/combined-R[B]-Q[v5+trait_ocean_v1]/`
  (no `-qm_*` suffix since override is off). This is intentional — the
  naming convention, enforced in `_combined_run_id()`, tags only when the
  questionnaire model differs from the rollout model.

Upload it so the next agent after you doesn't have to re-run:

```python
from src_dev.utils.hf_hub import upload_folder_to_dataset_repo, login_from_env
from pathlib import Path
login_from_env()

    local_dir=Path("scratch/psychometric_fa/combined-R[B]-Q[v5+trait_ocean_v1]"),
    repo_id="persona-shattering-lasr/psychometric-fa-runs",
    path_in_repo="combined/combined-R[B]-Q[v5+trait_ocean_v1]",
    commit_message="sid/psychometric-refactor upload Llama-3.1-8B combined FA+validation on B rollouts",
    ignore_patterns=[],

```

### 3. Run response-level comparison

```bash
uv run python -m scripts_dev.psychometric_assessment.compare_models
```

The script configuration lives at the top of `compare_models.py`:

- `RUNS`: list of `(model_label, questionnaire, run_id)` entries to compare.
  Already populated with Llama + Qwen3-8B + Qwen2.5-7B for v5, and Llama +
  Qwen2.5-7B for trait_ocean_v1.
- `QUESTIONNAIRES_TO_COMPARE`: which questionnaires to produce comparisons
  for (default: `["v5", "trait_ocean_v1"]`).
- `TAG`: subdirectory name under `scratch/psychometric_comparison/`.

Outputs:
- `overview_{questionnaire}_{model_a}_vs_{model_b}.png` — the main 7-panel
  summary figure per model pair.
- `agreement_{q}.csv`, `marginals_{q}.csv`, `summary_{q}.csv` — per-item and
  matrix-level stats.
- Individual plots: `hists_*.png`, `joint_heatmap_*.png`,
  `persona_scatter_*.png`, `agreement_bars_*.png`.

### 4. Run cross-model Tucker's congruence

Once both Llama and Qwen2.5 combined FA dirs are on disk:

```python
# In scripts_dev/psychometric_assessment/cross_model_factor_congruence.py,
# uncomment/update the llama entry in RUNS:
RUNS = [
    FaRun(
        label="qwen2.5-7b",
        combined_dir=Path("scratch/psychometric_fa/combined-R[B]-Q[v5+trait_ocean_v1]-qm_qwen257binstruct"),
        n_factors=7,
        rotation="oblimin",
    ),
    FaRun(
        label="llama-3.1-8b",
        combined_dir=Path("scratch/psychometric_fa/combined-R[B]-Q[v5+trait_ocean_v1]"),
        n_factors=7,
        rotation="oblimin",
    ),
]
```

Then:

```bash
uv run python -m scripts_dev.psychometric_assessment.cross_model_factor_congruence
```

Produces, under `scratch/psychometric_comparison/B_cross_model_congruence/<pair>/`:
- `congruence_matrix.csv` — full KA×KB φ matrix.
- `congruence_heatmap.png` — heatmap with Hungarian-matched cells outlined.
- `match_report.md` — matched pairs ranked by |φ|, interpretation
  (`≥0.95 equal`, `≥0.85 fair`, `<0.85 poor`), and top-loading items for each
  matched factor (sign-aligned to the A side).

The script is rotation-aware: oblimin ↔ oblimin, varimax ↔ varimax. To
compare both, run it twice with different `rotation=` in the `FaRun`
entries.

## Pipeline architecture (short reference)

Main driver: `scripts_dev/unsupervised_embeddings/psychometric_rollout_fa.py`

Stage flow:
1. `rollouts` — multi-turn persona generation (cached on HF via
   `rollouts-…` path)
2. `questionnaire` — administer Likert / FC / trait_mcq items at the N+1th
   turn of each rollout; produces the persona×item `response_matrix.npy`
   (cached on HF via `questionnaire-…` path). Cross-model runs tag the
   run-id with `-qm_<slug>`. Stage 2 config includes a `max_context_tokens`
   field and filter — populated automatically via `QUESTIONNAIRE_MAX_CONTEXT_TOKENS`
   for Qwen2.5 (32 768) to drop rollouts whose
   `tokens + max_new_tokens + retry_overhead + buffer` would overflow the
   model's native context.
3. `factor_analysis` — PAF + oblimin + varimax on the combined (multi-pair)
   matrix; writes `fa_7_principal_{rotation}.npz` into `combined-R[...]-
   Q[...]/factor_analysis/raw/`. Not auto-uploaded.
4. `validation` — shuffle/item_holdout/stability_sweep/bootstrap/split-half/
   k-sensitivity/persona-item-cv/… writes under `.../validation/`. Not
   auto-uploaded.
5. (Optional) `labeling` — LLM-labels for each factor. Off by default in
   `STAGES_TO_RUN`.

Cross-model knobs (all in `psychometric_rollout_fa.py`):
- `CROSS_MODEL_QUESTIONNAIRE` — bool; when True, populates overrides below.
- `QUESTIONNAIRE_MODEL_OVERRIDE` — e.g. `"Qwen/Qwen2.5-7B-Instruct"`.
- `QUESTIONNAIRE_MAX_CONTEXT_TOKENS` — e.g. 32768 for Qwen2.5.
- `QUESTIONNAIRE_CONTEXT_BUFFER_TOKENS` — default 1024.

When `QUESTIONNAIRE_MODEL_OVERRIDE` is set and differs from the rollout
model, both `_questionnaire_run_id` and `_combined_run_id` append
`-qm_<slug>` — so cached Stage-2 / Stage-3 artifacts don't collide with the
rollout-model run.

## Known quirks / gotchas

- **Combined FA dirs are not auto-hydrated.** The pipeline's caching logic
  is per-pair (Stage 2) only. Manual `hydrate_dataset_subtree` is needed
  (see snippet above) or just let Stage 3 re-run locally.
- **trait_ocean_v1's response matrix is continuous floats (0..1)**, unlike
  v5's discrete 1..5 integers. `compare_models.py` detects this via a
  small-unique-values heuristic and switches its plots to 2-D histograms /
  KDEs. If you add a new questionnaire with another encoding, verify the
  heuristic still fires correctly.
- **`QUESTIONNAIRE_MODEL_OVERRIDE` mutates globals via `_activate_rollout`.**
  Don't try to run multiple model overrides in a single script invocation —
  the run-id machinery assumes one override per session. Instead, do one
  pass per model.
- **One persona was dropped** by the context filter on Qwen2.5 runs
  (`sample_df6e45b2c0d3ba37aaa91c72`, 66 862 tokens — a PowerPoint VBA-macro
  truncation loop, the assistant regenerated the whole macro 13 times).
  All cross-model comparisons are run on the 2499-persona intersection.
- **Horn's PA suggests k=23** for the combined v5+trait_ocean_v1 matrix,
  but we force k=7 via `FA_N_FACTORS_OVERRIDE = 7` in the script to match
  the OCEAN-shaped prior expectation. `k_sensitivity` and `persona_item_cv`
  validation tests FAIL under this choice. The 1-SE rule on CV-k prefers
  k=11. Worth revisiting if factor interpretability is the goal.
- **Qwen3-8B lacks trait_ocean_v1.** Only the Llama↔Qwen2.5 trait_mcq
  comparison is populated. Generating Qwen3-8B's trait responses would just
  mean running the pipeline with `QUESTIONNAIRE_MODEL_OVERRIDE = "Qwen/Qwen3-8B"`,
  `PAIRS = [("B", "trait_ocean_v1")]`.

## If something breaks

- Stage 2 fails with OOM on a new machine: the Qwen2.5 vLLM run needs ~24 GB
  GPU to hold an `max_model_len=25_472` KV cache at bf16. On an RTX 5090
  (32 GB) it was fine. If your new machine is smaller, either lower
  `QUESTIONNAIRE_VLLM_GPU_MEMORY_UTILIZATION` or `QUESTIONNAIRE_MAX_CONTEXT_TOKENS`.
- Stage 3 FA crashes with near-singular covariance warnings: bump
  `MIN_ITEM_VARIANCE` or drop bimodal trait_mcq columns before FA.
- Tucker's congruence reports mostly `<0.85`: expected given Llama vs
  Qwen2.5's response-shape differences. The factor-scale question is still
  worth examining item-by-item via the `match_report.md`.
