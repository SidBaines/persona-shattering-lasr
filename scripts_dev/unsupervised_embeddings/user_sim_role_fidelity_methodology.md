# User-Simulator Role-Fidelity Audit ‚ÄĒ Methodology

Reference documentation for the audit of user-simulator role fidelity in multi-turn rollouts generated by `scripts_dev/unsupervised_embeddings/psychometric_rollout_fa.py`. This doc is designed to be sufficient to (a) reproduce the audit on a new rollout run, and (b) write up the method in the paper's unsupervised section.

**Associated artifacts:**
- Findings report: `review_user_sim.md` (top level of repo)
- Analysis scripts: `scratch/user_sim_review/phase1_heuristics.py`, `scratch/user_sim_review/phase2_sample_batches.py`
- Judge prompt template: `scratch/user_sim_review/judge_prompt_template.md`

## 1. Motivation

Psychometric-factor-analysis pipelines assume the "user" side of a rollout is a consistent stand-in for a human. When a user-simulator LLM drifts into *assistant* behavior (giving advice, producing structured answers, offering help, breaking into AI self-reference) the conversation no longer represents a user's expression of traits ‚ÄĒ it represents a model talking to itself. Role-fidelity is therefore a precondition for the FA's construct validity.

The system prompt used for the user-simulator in this project (see `scripts_dev/unsupervised_embeddings/user_simulator_archetype_prompts.py`, `CORE_RULES`) explicitly forbids helper behaviors, but prior inspection suggested instructions were not uniformly followed. This audit quantifies fidelity and characterizes failure modes.

## 3. Two-phase pipeline

### Phase 1 ‚ÄĒ heuristic sweep (cheap, full-corpus)

A pure-Python regex/structural sweep over every user-sim turn. Purpose: establish a baseline rate of obvious formatting tells; stratify Phase 2 sampling toward likely-interesting conversations. Does *not* catch semantic slippage.

**Heuristics** (implemented in `scratch/user_sim_review/phase1_heuristics.py`):

| flag | trigger | weight |
|---|---|---|
| `bullet_list` | ‚Č•2 lines matching `^\s*[-*‚ÄĘ]\s+\S` | 2.0 |
| `numbered_list` | ‚Č•2 lines matching `^\s*\d+[.)]\s+\S` | 2.0 |
| `md_header` | any `^\s*#{1,6}\s+\S` | 2.0 |
| `bold_header` | any `^\s*\*\*[^*\n]{2,80}:\*\*` | 2.0 |
| `code_fence` | presence of ``` ``` ``` | 1.0 |
| `assistant_opener` | starts with "Certainly!", "Absolutely!", "Of course!", "Sure!", "Great question", "Happy/Glad to help", "I'd be happy to" | 3.0 |
| `heres_framing` | "here's a/the/some/how/what/my/a few" or "here are ‚Ä¶" | 1.5 |
| `step_label` | `step\s*\d+[:.]` | 2.0 |
| `firstly_secondly` | `firstly / secondly / thirdly / first, / second, / third, / finally,` | 1.0 |
| `offer_help` | "Would you like me to‚Ä¶", "Shall I‚Ä¶", "I can help you‚Ä¶", "let me know if you‚Ä¶", "feel free to ask/reach/let", "If you'd like, I can‚Ä¶", "I hope this/that helps", "hope this helps", "I'm happy to", "Happy to clarify/explain/help" | 3.0 |
| `ai_self` | "As an AI", "I'm an AI", "my training data", "my knowledge cutoff", "I'm a language model" | 4.0 |
| `summary_framing` | "to summarize/summarise", "in summary", "in conclusion", "to recap", "overall,", "key points/takeaways", "here's a summary/recap/breakdown", "breakdown:" | 2.0 |
| `disclaimer` | "please note that", "it's worth/important noting", "keep in mind that", "a gentle reminder", "disclaimer:" | 1.5 |

Weighted sum is `slip_score`. Binary `any_flag` = any heuristic triggered.

**Preceding-assistant-structured feature:** per user-sim turn, the immediately preceding assistant message is also checked against the same bullet/numbered/header patterns. Lets us test the mirroring hypothesis directly in Phase 1.

**Output:**
- `scratch/user_sim_review/phase1_per_turn.csv` ‚ÄĒ one row per user-sim turn
- `scratch/user_sim_review/phase1_aggregates.json` ‚ÄĒ cross-tabs by archetype, scenario_category, turn_index

Run: `python3 scratch/user_sim_review/phase1_heuristics.py`

### Phase 2 ‚ÄĒ stratified LLM-judge sample

The formatting heuristics miss semantic failure modes (e.g. user gives advice in plain prose, user adopts advisor voice without bullets). Phase 2 applies a Sonnet-class LLM rubric-judge to a stratified sub-sample.

#### Sampling

Stratification: 3 conversations per (archetype √ó scenario_category) cell = 300 conversations across 100 cells.

Within each cell, bias toward heuristic-flagged conversations so the judge sees known-interesting cases alongside controls:
- 2 conversations with ‚Č•1 Phase-1-flagged turn, if available
- 1 clean conversation
- Top-up with anything available if the cell lacks enough flagged or clean conversations

`random.seed(432)` for reproducibility. Implemented in `scratch/user_sim_review/phase2_sample_batches.py`. Outputs 10 batches of 30 conversations each (`scratch/user_sim_review/phase2_batches/batch_NN.json`).

#### Rubric (per user-sim turn)

| field | domain | description |
|---|---|---|
| `role_fidelity` | 1‚Äď5 | 1=fully assistant, 5=fully user |
| `failure_mode` | enum | one of `none / offers_help / structured_answer / solution_first / summarizing / meta_commentary / hedging_disclaimer / ai_identity / tone_flip / other` |
| `failure_note` | string or null | required when fidelity ‚Č§ 3; a quote or paraphrase of the tell |
| `preceded_by_structured_assistant` | bool | true iff the immediately preceding assistant message uses bullets/bold headers/numbered lists |

Per-conversation aggregation:

| field | description |
|---|---|
| `min_fidelity` | lowest turn score |
| `first_slip_turn_index` | lowest turn index where fidelity ‚Č§ 3, or null |
| `worst_slip_turn_index` | turn index where fidelity was lowest (ties: earliest) |
| `recovers_after_slip` | True if any later turn has fidelity ‚Č• 4 after first slip; null if no slip |
| `sticky_slip` | True if all turns after first slip remain ‚Č§ 3; null if no slip |
| `conversation_notes` | 1‚Äď2 sentences on overall pattern |

#### Failure-mode taxonomy

- **offers_help**: user offers to help or teach the assistant.
- **structured_answer**: user gives a formatted answer (bullets / headers / numbered list) as if responding to a prompt from the assistant.
- **solution_first**: user proposes a structured *solution* to their own problem rather than continuing to engage as the one with the problem (often shows as second-person self-address).
- **summarizing**: user recaps the conversation or prior messages as a meta-summary.
- **meta_commentary**: user speaks about the conversation / the models as a third party.
- **hedging_disclaimer**: "it's worth noting‚Ä¶", "please keep in mind‚Ä¶" in user voice.
- **ai_identity**: user refers to itself as an AI / language model / "trained to do X".
- **tone_flip**: tone shifts to formal / professional / helper register without scenario justification.
- **other**: describe in `failure_note`.

#### Calibration rules given to the judge

- **Do not penalize archetype-appropriate structure.** A `precise` user naturally being terse-and-listy is not a slip; the signal is when the user takes the *helper's* advisory stance.
- **Do not penalize short/blunt answers.** "Yeah." is fine.
- **Code blocks can be user-appropriate** when the user shares their own code or an error; only a slip when the user is giving a teachy answer.
- **Mirroring isn't automatic slip.** If the assistant used headers, the user can still respond casually without them. Only flag if the user starts structuring.
- **Err toward fidelity=5 on ambiguous cases.** False negatives are less costly than noise for this analysis.

Full rubric lives at `scratch/user_sim_review/judge_prompt_template.md`.

#### Dispatch

10 Sonnet subagents run in parallel, one per batch file. Each subagent is given:

1. The path to `judge_prompt_template.md`.
2. The path to its `batch_NN.json` input file.
3. The path at which to write its JSON output.
4. A one-paragraph calibration reminder.

Per-subagent prompt (parameterized by batch index):

```
You are judging user-simulator role-fidelity in a batch of 30 research conversations.

STEP 1: Read the full rubric at:
scratch/user_sim_review/judge_prompt_template.md

STEP 2: Read the input batch at:
scratch/user_sim_review/phase2_batches/batch_{BATCH_ID}.json

STEP 3: Following the rubric exactly, produce per-turn judgments
(role_fidelity 1-5, failure_mode enum, failure_note,
preceded_by_structured_assistant) for every user-simulator turn in every
conversation, plus per-conversation summaries (min_fidelity,
first_slip_turn_index, worst_slip_turn_index, recovers_after_slip,
sticky_slip, conversation_notes).

STEP 4: Write the result as a single JSON object with shape
`{"batch_id": "{BATCH_ID}", "conversations": [...]}` to
scratch/user_sim_review/phase2_judgments/batch_{BATCH_ID}.json

Do NOT return the JSON in your reply ‚ÄĒ write to file and reply with one
line: `wrote N conversations to <path>`.

Judge every conversation. Do not skip or return partial results.

Calibration: the archetypes (blunt, precise, etc.) are legit user styles
‚ÄĒ don't confuse archetype-driven structure with helper-mode slippage. The
slip signal is when the user adopts a HELPER's voice (giving advice,
offering to help, summarizing, structured answers to their own
questions). Err toward fidelity=5 on ambiguous cases.
```

## 4. Known gotchas for reproduction

- **Subagent write permissions.** In the reference run, 3 of 10 subagents were blocked from writing output files (`Write` and `Bash` denied in subagent scope) and returned unstructured prose summaries or nothing. Before re-running, verify subagents have `Write` (or `Bash`) tool access via `~/.claude/settings.json` / `.claude/settings.local.json`. A robust fallback is to have each subagent **return the full JSON as its final reply** (not a file write); the orchestrator parses and writes.
- **Context budget.** Each batch file is ~600 KB (~150K tokens) for 30 √ó 10-turn conversations. Reducing batch size (e.g., 15 convs / batch, 20 batches) lowers per-subagent load at the cost of more orchestration.
- **Early-termination conversations.** Some rollouts in the reference corpus terminated before turn 9 (2,000 √ó 10 = 20,000 expected turns, only 17,895 observed). Judges must tolerate <10 turns per conversation.
- **Scenario-id / archetype extraction.** Both are encoded in the `user_prompt_template` string (`sc_{archetype}_{scenario_id}_sample_{hash}_r{replicate}_{hash}`). `scenario_id` itself contains underscores, so split on `_sample_`.

## 5. Suggested extensions

- **Close the calibration loop** by running the same rubric twice with different LLM judges (e.g., Sonnet + Opus) and computing inter-rater agreement on `role_fidelity`.
- **Turn-granular logistic regression** with features `{turn_index, preceded_by_structured_assistant, archetype dummies, scenario_category dummies, previous_turn_was_slip}` to formally quantify per-factor slip probability.
- **Content-level filter applied upstream**: any conversation where max `slip_score` > threshold (Phase 1) or min `role_fidelity` ‚Č§ 3 (Phase 2) is excluded from downstream FA; compare FA loadings / trait structure with and without filtering.
- **A/B the `UserSimulatorConfig.turn_reminder` field** (currently None). Re-run a 200-conversation subset with and without a per-turn "stay in user character" reminder; compare Phase 1 and Phase 2 rates.
- **Cross-simulator comparison**: repeat with `openai/gpt-4o-mini` or another user-simulator backend to establish whether the slip patterns are GLM-specific or generic to LLM user-simulators at this scale.

## 6. Paper-write-up pointers

The findings motivate a short methods-section paragraph on **construct validity of the user side of the rollout corpus**. Key points that belong in the paper:

1. Unsupervised factor analysis on rollout-derived embeddings assumes user-side content is representative of a (psychometrically coherent) human partner.
2. In practice, with this user-simulator, ~20% of conversations contain at least one role-fidelity failure. The dominant failure modes are *learner-becomes-teacher*, *self-coaching*, and *AI-identity break* ‚ÄĒ all of which introduce assistant-mode variance into the user-side corpus.
3. Mitigation in this work (recommended): filter conversations by LLM-judge `min_fidelity` before FA; report sensitivity of the reported factor structure to the filter threshold.
4. The failure modes are not uniformly distributed across archetypes or scenario categories: mentioning this in the paper both (a) documents a real methodological limit and (b) motivates the archetype / scenario ablations in supplementary material.
